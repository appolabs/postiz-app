<objective>
Switch Next.js to standalone output mode to ship only traced dependencies instead of full node_modules.

Purpose: In the current setup, `next start` requires the entire `node_modules` tree (~800MB+) at runtime. Standalone mode uses `@vercel/nft` to trace only the files the frontend actually imports, producing a self-contained `apps/frontend/.next/standalone/` directory with a `server.js` entry point. This reduces runtime memory by ~200-300MB because Node loads fewer modules and the OS maps fewer files.

Output: Frontend builds to `.next/standalone/`, starts via `node server.js` instead of `next start`, static assets and public files are correctly served.

## How standalone mode works (Next.js 14.2)

When `output: 'standalone'` is set in `next.config.js`:

1. `next build` produces `.next/standalone/` â€” a minimal copy of the app with only traced dependencies.
2. The standalone directory mirrors the monorepo root structure:
   ```
   .next/standalone/
   â”śâ”€â”€ node_modules/          (traced subset, ~50MB vs ~800MB)
   â”śâ”€â”€ apps/
   â”‚   â””â”€â”€ frontend/
   â”‚       â”śâ”€â”€ .next/
   â”‚       â”‚   â””â”€â”€ server/    (server chunks)
   â”‚       â””â”€â”€ server.js      (entry point)
   â”śâ”€â”€ libraries/             (traced library files)
   â””â”€â”€ package.json
   ```
3. The standalone server does NOT serve static files or the `public/` directory.
   These must be copied manually after build:
   - `apps/frontend/.next/static/` -> `apps/frontend/.next/standalone/apps/frontend/.next/static/`
   - `apps/frontend/public/` -> `apps/frontend/.next/standalone/apps/frontend/public/`
4. The server starts with: `node apps/frontend/.next/standalone/apps/frontend/server.js`
5. Environment variables: `PORT` and `HOSTNAME` control the listen address.
6. Dev mode (`next dev`) is completely unaffected â€” standalone only applies to production builds.

## Current setup

- **Build**: `next build` in `apps/frontend/` (invoked via `pnpm --filter ./apps/frontend run build`)
- **Start (prod)**: `dotenv -e ../../.env -- next start -p 4200` (via pm2 in Docker)
- **Start (Docker)**: pm2 runs `pnpm start` in the frontend workspace, which calls `next start -p 4200`
- **Nginx**: Proxies `localhost:4200` to the user, so the port must remain 4200.
- **Public dir**: 3.2MB of static assets (favicon, logos, platform icons, auth backgrounds). Referenced via `<link rel="icon" href="/favicon.ico">` and `<img>` tags in components.
- **Sentry**: `withSentryConfig()` wraps the Next config. Sentry's webpack plugin runs during build and is compatible with standalone. Sourcemap upload happens at build time. The Sentry instrumentation files (`instrumentation.ts`, `sentry.server.config.ts`, `sentry.edge.config.ts`) are bundled into the server output and work with standalone.
- **Image optimization**: `next/image` with remote patterns configured. Standalone includes the sharp module if installed (it is â€” `sharp` is in root dependencies).

## Monorepo path structure

In a pnpm workspaces monorepo, Next.js standalone traces from the **monorepo root** (detected via `outputFileTracingRoot` or automatically). The standalone output preserves the full path structure relative to root. When the frontend is at `apps/frontend/`, the server entry point lands at `.next/standalone/apps/frontend/server.js`, not `.next/standalone/server.js`.

The `outputFileTracingRoot` should be set to `../../` (two levels up from `apps/frontend/` to the monorepo root) so that shared libraries under `libraries/` are properly traced.

<risks>

1. **sharp module tracing**: The `sharp` native module must be traced correctly for `next/image` optimization. Next.js 14.2 handles this automatically when sharp is installed. If images break, the fallback is `images: { unoptimized: true }` â€” but this should not be needed.

2. **Sentry sourcemap paths**: Sentry's `sourcemaps.assets` patterns reference `.next/static/**` and `.next/server/**`. These paths are relative to the build directory and are processed at build time, not runtime. Standalone does not affect this.

3. **Environment variables at runtime**: Standalone server.js reads env vars at startup, same as `next start`. The `dotenv -e ../../.env` prefix still works because the server.js is started from the `apps/frontend/` working directory (pm2 runs in workspace context). However, note that the standalone server.js file is at `.next/standalone/apps/frontend/server.js` â€” when Node resolves `../../.env` from *that* path, it would look in `.next/standalone/.env`, not the repo root. This is why we use `dotenv` CLI to load env vars *before* starting node, which works correctly regardless of server.js location.

4. **Monorepo workspace resolution**: pnpm's `--filter` commands run in the workspace directory (`apps/frontend/`). The `postbuild` script uses relative paths from that directory. This is correct.

5. **Middleware**: If Next.js middleware exists, it is included in standalone output automatically. No special handling needed. (No middleware files were found in the frontend.)

6. **i18n / rewrites / redirects**: These are compiled into the standalone server at build time. The existing redirects and rewrites in next.config.js will continue to work.

</risks>
