# Phase 07: Defer AI/Chat Module Loading

## 1. Context

### Current Boot-Time Loading

The backend (`apps/backend/src/main.ts`) does two things that force all AI dependencies into memory at startup:

**A. ChatModule loaded eagerly via AppModule**

`app.module.ts` imports `ChatModule` which is `@Global()`. At module initialization NestJS resolves all providers:

```
ChatModule
  -> MastraService         (imports @mastra/core/mastra, @mastra/core/logger)
  -> LoadToolsService      (imports @mastra/core/agent, @ai-sdk/openai, @mastra/memory)
  -> 8 tool classes         (each imports @mastra/core/tools)
```

The `mastra.store.ts` file is imported by `MastraService` and creates a `PostgresStore` instance at module evaluation time:
```ts
// mastra.store.ts -- executes on import
export const pStore = new PostgresStore({ connectionString: process.env.DATABASE_URL });
```

This means `@mastra/pg` (PostgresStore, PgVector) is loaded as soon as any file in the chat directory is imported.

**B. `startMcp(app)` called synchronously at boot**

`main.ts` line 47 calls `await startMcp(app)` which:
1. Resolves `MastraService` from the DI container
2. Calls `mastraService.mastra()` which instantiates `Mastra` + `Agent` + `Memory`
3. Gets the agent's tools
4. Creates an `MCPServer` from `@mastra/mcp`
5. Mounts two Express middleware handlers on `/mcp` and `/mcp/:id`

This means **even if no user ever uses AI/chat**, the full Mastra, Agent, Memory, and MCP server are initialized at boot, blocking startup.

**C. AgentModule loaded eagerly via AppModule**

`AgentModule` provides `AgentGraphService` and `AgentGraphInsertService`, both of which have top-level module scope instantiation of LangChain models:

```ts
// agent.graph.service.ts -- executes at module evaluation
const model = new ChatOpenAI({ ... });
const dalle = new DallEAPIWrapper({ ... });
const tools = [new TavilySearchResults(...)];
const toolNode = new ToolNode(tools);
```

```ts
// agent.graph.insert.service.ts -- executes at module evaluation
const model = new ChatOpenAI({ ... });
```

These pull in `@langchain/openai`, `@langchain/langgraph`, `@langchain/core`, `@langchain/community` at module load time.

**D. AutopostService in DatabaseModule imports LangChain**

`autopost.service.ts` is part of `DatabaseModule` (the core module) and has top-level LangChain instantiation:
```ts
const model = new ChatOpenAI({ ... });
const dalle = new DallEAPIWrapper({ ... });
```

This is the hardest to defer because `DatabaseModule` is the core database module used everywhere.

**E. OpenaiService in DatabaseModule imports `openai` SDK**

```ts
// openai.service.ts -- executes at module evaluation
const openai = new OpenAI({ apiKey: process.env.OPENAI_API_KEY || 'sk-proj-' });
```

## 2. Strategy

### Two-Tier Approach

The AI dependencies split into two tiers based on how deeply they're embedded:

**Tier 1 (ChatModule + MCP + AgentModule) -- Full lazy-load via NestJS LazyModuleLoader**

These modules are self-contained and only accessed through specific controllers/routes. They can be converted to lazy-loaded modules using NestJS's built-in `LazyModuleLoader`.

- Remove `ChatModule` and `AgentModule` from `AppModule.imports`
- Remove `startMcp(app)` from `main.ts`
- Lazy-load these modules on first request to their respective endpoints
- Defer MCP server setup to first `/mcp` request

**Tier 2 (AutopostService + OpenaiService in DatabaseModule) -- Defer heavy imports within services**

These services live in `DatabaseModule` which cannot be lazy-loaded (it's the core module). Instead, move the heavy top-level instantiations behind lazy initialization patterns:

- Replace top-level `const model = new ChatOpenAI(...)` with factory functions called on first use
- Replace top-level `const openai = new OpenAI(...)` with lazy getter
- Use dynamic `import()` for the SDK packages themselves

### Why Not NestJS `forwardRef` or `DynamicModule`?

`LazyModuleLoader` is the officially supported NestJS pattern for deferring module initialization. It:
- Defers all provider instantiation until `lazyModuleLoader.load()`
- Defers all static `import` resolution until the module is first loaded
- Works with `@Global()` modules (but we'll remove `@Global()` from deferred modules)

### 3.5. Create Lazy Provider Service for CopilotController

**New file: `libraries/nestjs-libraries/src/chat/lazy.chat.provider.ts`**

A thin service registered in `AppModule` (or `ApiModule`) that uses `LazyModuleLoader` to load `ChatModule` on demand:

```ts
import { Injectable } from '@nestjs/common';
import { LazyModuleLoader } from '@nestjs/core';

@Injectable()
export class LazyChatProvider {
  private _mastraService: any = null;

  constructor(private lazyModuleLoader: LazyModuleLoader) {}

  async getMastraService() {
    if (!this._mastraService) {
      const { ChatModule } = await import('./chat.module');
      const moduleRef = await this.lazyModuleLoader.load(() => ChatModule);
      const { MastraService } = await import('./mastra.service');
      this._mastraService = moduleRef.get(MastraService);
    }
    return this._mastraService;
  }
}
```

## 5. Fallback: What Happens If AI Features Are Never Used

If no user ever triggers any AI endpoint:
- **ChatModule** is never loaded. `@mastra/*`, `@ai-sdk/openai`, `@mastra/mcp` stay unloaded.
- **AgentModule** is never loaded. `@langchain/*` stays unloaded.
- **OpenaiService** never imports `openai` SDK. The service class itself is still registered in DatabaseModule (it's a lightweight shell with no top-level imports), but the SDK is never loaded.
- **AutopostService** class is still registered in DatabaseModule, but LangChain modules are never dynamically imported unless `startAutopost()` runs.
- **MCP middleware** responds to no requests; the handler function exists but never triggers initialization.
- **Net effect**: ~55-110 MB saved. The backend boots with zero AI/ML library overhead.

## 7. Risks

### 7.1. First-Request Latency Spike

The first request to any AI endpoint will be slower (~200-500ms) because it triggers dynamic imports and module initialization.

**Mitigation**: Acceptable tradeoff. AI operations (LLM API calls) already take 1-10 seconds. A 200-500ms import overhead is negligible in that context. Users see the chat loading indicator during this time.

### 7.2. NestJS LazyModuleLoader Caveats

`LazyModuleLoader` has known behaviors:
- The lazily loaded module's providers are NOT available globally even if `@Global()` was set
- The module must import its own dependencies explicitly
- Circular dependencies between the lazy module and eagerly loaded modules will fail

**Mitigation**: Remove `@Global()` from deferred modules. The lazy provider wrappers (`LazyChatProvider`, `LazyAgentProvider`) handle the resolution. Chat tools already get their dependencies (IntegrationService, PostsService, etc.) via `ModuleRef`, so they need access to the main DI container. Verify that `LazyModuleLoader.load()` returns a module ref that can resolve providers from parent modules.

**Alternative if LazyModuleLoader doesn't work well**: Skip NestJS lazy loading entirely. Instead, keep ChatModule/AgentModule in AppModule but change ALL their internal imports to dynamic. The module shells would be empty at boot -- providers register but don't import heavy SDKs until their methods are called. This is the Tier 2 approach applied universally. It's more work but avoids NestJS-specific edge cases.

### 7.3. Dynamic Import in Compiled Output

The backend compiles with `tsc` to CommonJS. Dynamic `import()` in CommonJS is supported in Node >= 14, but:
- TypeScript may compile `await import('x')` differently depending on `module` setting
- The `tsconfig` must support dynamic import syntax

**Mitigation**: Check `tsconfig.base.json` and `apps/backend/tsconfig.json` to confirm `module` is set to a value that supports dynamic import (e.g., `commonjs` with `esModuleInterop`, or `nodenext`). If needed, use `require()` instead of `import()` for CommonJS targets, but this doesn't provide tree-shaking.

### 7.4. Race Conditions on Concurrent First Requests

Multiple simultaneous requests to `/mcp` or `/copilot/agent` could trigger parallel initialization.

**Mitigation**: Use the promise-caching pattern in all lazy initializers:

```ts
let initPromise: Promise<void> | null = null;
async function ensureInitialized() {
  if (!initPromise) {
    initPromise = doInit();
  }
  return initPromise;
}
```

This ensures only one initialization runs; concurrent callers await the same promise.

### 7.5. AutopostService Stays in DatabaseModule

`AutopostService` cannot be moved out of `DatabaseModule` without a large refactoring effort (it depends on PostsService, IntegrationService, etc. from the same module). The Tier 2 approach (dynamic imports within the service) is the pragmatic solution.

**Risk**: If any other file in `DatabaseModule` happens to statically import from `autopost.service.ts` for a type, the LangChain modules would still load at boot.

**Mitigation**: Verify that `autopost.service.ts` is only imported by the DatabaseModule provider registration (which uses the class token, not the module body). TypeScript type-only imports (`import type { ... }`) don't trigger module evaluation.

### 7.6. `mastra.store.ts` Creates PG Connection at Import

Currently, `new PostgresStore({ connectionString })` likely opens a connection pool to PostgreSQL at creation time. Deferring this means the PG connection for Mastra storage is only created when AI features are first used.

**Risk**: None -- this is actually desirable. Fewer idle PG connections for non-AI deployments.

### 7.7. Module Initialization Failures

If AI modules fail to initialize (e.g., missing OPENAI_API_KEY, PostgreSQL connection error for Mastra store), the error will occur at first request rather than at boot.

**Risk**: Harder to debug during deployment since the error won't appear in startup logs.

**Mitigation**: The existing code already has runtime checks (e.g., `if (process.env.OPENAI_API_KEY === undefined)` in CopilotController). These continue to work. Add a startup log message: `"AI modules will be loaded on first request"` so operators know to check request logs for AI initialization errors.
