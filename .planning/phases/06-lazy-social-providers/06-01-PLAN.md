# Phase 06: Lazy Social Provider Instantiation

## 1. Context

### Current Pattern

`integration.manager.ts` eagerly instantiates all 31 social providers at module evaluation time via a top-level `export const socialIntegrationList` array:

```ts
export const socialIntegrationList: Array<SocialAbstract & SocialProvider> = [
  new XProvider(),
  new LinkedinProvider(),
  // ... 29 more
];
```

Every `import` of `integration.manager.ts` -- whether from the backend, orchestrator, or chat tools -- triggers all 31 `new Provider()` calls and, critically, **all 31 top-level `import` statements** that pull in each provider's transitive dependency tree (`twitter-api-v2`, `sharp`, `@atproto/*`, `nostr-tools`, `image-to-pdf`, etc.).

### Memory Impact

Each provider file imports platform-specific SDKs at the top level. Even though the provider constructors are cheap (no constructors, no args, just property declarations), the **module loading** is expensive:
- Node.js parses, compiles, and caches every imported module
- SDK dependencies like `twitter-api-v2`, `sharp`, `@atproto/*`, `nostr-tools`, etc. each allocate memory for their module graphs
- Estimated ~2-5 MB per provider (SDK + transitive deps), totaling ~50-150 MB for 31 providers
- A typical deployment only uses 3-5 providers, so ~80-90% of this memory is wasted

### Provider Characteristics

- **33 provider files** in `libraries/nestjs-libraries/src/integrations/social/*.provider.ts`
- **31 active** (one commented out: `MastodonCustomProvider`)
- **Zero constructors** -- all providers use only property initializers (`identifier`, `name`, `scopes`, etc.)
- **Stateless** -- no shared mutable state between calls; each method receives tokens/IDs as arguments
- All extend `SocialAbstract` and implement `SocialProvider`
- Some use `@Plug`, `@PostPlug`, `@Rules` decorators that store metadata via `Reflect.defineMetadata`

## 2. Strategy: Lazy Map with Dynamic `import()`

Replace the eagerly-instantiated array with a **lazy registry** that uses dynamic `import()` to load each provider on first access. The registry maps `identifier` strings to factory functions that return a `Promise<SocialAbstract & SocialProvider>`.

Key design decisions:
- **Dynamic `import()` over static `import`**: This is the only way to defer Node.js module loading. Require statements and static imports both execute at module evaluation time.
- **Cache after first load**: Each provider is instantiated once and cached. Subsequent lookups return the cached instance synchronously (wrapped in a resolved promise).
- **Preserve the `socialIntegrationList` export** for backward compatibility, but change it from a pre-populated array to a function/accessor that returns only loaded providers (or a full descriptor list for metadata-only operations).
- **Metadata registry**: Provider metadata needed at startup (identifier, name, toolTip, editor, maxConcurrentJob, etc.) must be available without loading the full provider. Store this as a static descriptor array.

## 3. Changes

### 3.1. `integration.manager.ts` -- Core Changes

#### 3.1.1. Replace top-level imports with a descriptor + factory registry

Remove all 31 static imports. Replace with:

```ts
// Static metadata for each provider -- no module loading required
interface ProviderDescriptor {
  identifier: string;
  name: string;
  maxConcurrentJob: number;
  // factory returns the module path for dynamic import
  modulePath: string;
  // class name exported from the module
  className: string;
}

const providerRegistry: ProviderDescriptor[] = [
  { identifier: 'x', name: 'X', maxConcurrentJob: 1,
    modulePath: './social/x.provider', className: 'XProvider' },
  { identifier: 'linkedin', name: 'LinkedIn', maxConcurrentJob: 2,
    modulePath: './social/linkedin.provider', className: 'LinkedinProvider' },
  // ... all 31 providers
];
```

#### 3.1.2. Lazy instance cache

```ts
const providerCache = new Map<string, SocialAbstract & SocialProvider>();

async function loadProvider(identifier: string): Promise<SocialAbstract & SocialProvider> {
  const cached = providerCache.get(identifier);
  if (cached) return cached;

  const descriptor = providerRegistry.find(d => d.identifier === identifier);
  if (!descriptor) throw new Error(`Unknown provider: ${identifier}`);

  const mod = await import(descriptor.modulePath);
  const instance = new mod[descriptor.className]() as SocialAbstract & SocialProvider;
  providerCache.set(identifier, instance);
  return instance;
}

// Synchronous lookup for already-loaded providers (returns undefined if not loaded)
function getLoadedProvider(identifier: string): (SocialAbstract & SocialProvider) | undefined {
  return providerCache.get(identifier);
}
```

#### 3.1.3. Update `socialIntegrationList` export

The exported `socialIntegrationList` is accessed in two fundamentally different ways:

**A. Metadata-only iteration** (no provider methods called) -- used for listing available integrations, building identifier lists, extracting reflect metadata. These need the full provider instance because decorators like `@Plug`, `@PostPlug`, `@Rules` store metadata on the constructor prototype.

**B. Single-provider lookup** (provider methods called) -- used for auth, posting, analytics, token refresh. These only need one specific provider.

For (B), the `IntegrationManager.getSocialIntegration()` method becomes async.

For (A), the metadata-dependent methods (`getAllIntegrations`, `getAllTools`, `getAllRulesDescription`, `getAllPlugs`, `getInternalPlugs`, `getAllowedSocialsIntegrations`) have two options:
  - **Option 1**: Make them async and load all providers on demand (lazy but loads everything on first call to these endpoints)
  - **Option 2**: Extract decorator metadata into the static registry at build/startup time

**Chosen approach: Option 1** for simplicity. These methods are called infrequently (on page load, not per-post), and once loaded, providers stay cached. The first call to `getAllIntegrations()` triggers loading all providers, but subsequent calls are instant. This is acceptable because:
  - `getAllIntegrations()` is only called when the frontend loads the channel list
  - The orchestrator never calls these bulk methods
  - Once loaded, memory stays allocated -- but that only happens when a user actually uses the feature

However, a critical optimization: **the Temporal worker registration in `temporal.module.ts`** iterates `socialIntegrationList` to create per-provider task queues. This runs at startup and is synchronous. For this specific case, we use the static `providerRegistry` descriptors which contain `identifier` and `maxConcurrentJob` -- no dynamic import needed.

#### 3.1.4. Updated `IntegrationManager` class

```ts
@Injectable()
export class IntegrationManager {
  async getAllIntegrations() {
    const providers = await loadAllProviders();
    return {
      social: await Promise.all(
        providers.map(async (p) => ({
          name: p.name,
          identifier: p.identifier,
          // ... same as current
        }))
      ),
      article: [] as any[],
    };
  }

  // These become async:
  async getAllTools(): Promise<{ [key: string]: { description: string; dataSchema: any; methodName: string }[] }> { ... }
  async getAllRulesDescription(): Promise<{ [key: string]: string }> { ... }
  async getAllPlugs(): Promise<any[]> { ... }
  async getInternalPlugs(providerName: string): Promise<any> { ... }

  // This stays sync -- uses static registry
  getAllowedSocialsIntegrations(): string[] {
    return providerRegistry.map((p) => p.identifier);
  }

  // This becomes async
  async getSocialIntegration(integration: string): Promise<SocialProvider> {
    return loadProvider(integration);
  }
}
```

## 7. Risks

### 7.1. Dynamic `import()` Path Resolution

Dynamic `import()` with relative paths works in both CommonJS and ESM, but the path must be resolvable at runtime from the compiled output location. Since the backend uses `tsc` compilation and the relative paths between `integration.manager.ts` and `social/*.provider.ts` are preserved in the output, this should work.

**Mitigation**: Use `path.join(__dirname, descriptor.modulePath)` if bare relative imports fail in the compiled output. Test with `pnpm run build:backend` before merging.

### 7.2. First-Request Latency

The first request that triggers a provider load will have a small delay (~10-50ms) for the dynamic import. This is negligible compared to the network calls these providers make (OAuth, API calls).

**Mitigation**: None needed. The delay is imperceptible.

### 7.3. Race Condition on Concurrent First Loads

Two concurrent requests for the same provider could trigger two `import()` calls before the cache is populated.

**Mitigation**: Use a pending-promise cache pattern:

```ts
const loadingPromises = new Map<string, Promise<SocialAbstract & SocialProvider>>();

async function loadProvider(identifier: string) {
  if (providerCache.has(identifier)) return providerCache.get(identifier)!;
  if (loadingPromises.has(identifier)) return loadingPromises.get(identifier)!;

  const promise = doLoad(identifier);
  loadingPromises.set(identifier, promise);
  try {
    const instance = await promise;
    providerCache.set(identifier, instance);
    return instance;
  } finally {
    loadingPromises.delete(identifier);
  }
}
```

### 7.4. Reflect Metadata Availability

Decorators like `@Plug`, `@PostPlug`, `@Rules` store metadata via `Reflect.defineMetadata`. This metadata is available as soon as the class is defined (during module evaluation). Since `loadProvider` triggers `import()` which evaluates the module, the metadata will be available by the time the instance is returned. The existing `import 'reflect-metadata'` at the top of `integration.manager.ts` ensures the polyfill is loaded first.

**Risk**: None -- decorator metadata is set at class definition time, not at instantiation time.

### 7.5. Missing Provider at Runtime

If a provider identifier from the database doesn't match any entry in `providerRegistry`, `loadProvider` will throw. This is the same behavior as the current `.find()!` pattern which returns `undefined` and crashes later.

**Mitigation**: `loadProvider` throws a clear error message: `Unknown provider: ${identifier}`. This is actually an improvement over the current silent `undefined` return.

### 7.6. Temporal Worker Registration

The temporal module reads `socialIntegrationList` synchronously at module registration time to create per-provider task queues. Switching to `socialIntegrationDescriptors` preserves this synchronous pattern -- no async loading needed, since we only read `identifier` and `maxConcurrentJob` from the static registry.

**Risk**: None -- the descriptor has the exact fields needed.

### 7.7. `integration.validation.tool.ts` Inline String

Line 32 builds a description string at tool creation time:
```ts
`platform identifier (${socialIntegrationList.map(p => p.identifier).join(', ')})`
```

This executes when the tool is created (at NestJS module init), not at request time. Replacing with `providerRegistry.map()` is a direct substitution.

**Risk**: None.
