  <action>
Modify both `getPosts()` and `getPostsList()` methods to filter by `integration.providerIdentifier` when `query.platform` is provided.

**Important:** The existing `customer` filter uses a spread pattern that overwrites the earlier `integration: { deletedAt: null }` condition. Fix this by building a single `integration` object that combines all conditions.

In `getPosts()` (around line 154-165), replace the current pattern:
```typescript
integration: {
  deletedAt: null,
},
deletedAt: null,
parentPostId: null,
...(query.customer
  ? {
      integration: {
        customerId: query.customer,
      },
    }
  : {}),
```

With a merged `integration` object:
```typescript
integration: {
  deletedAt: null,
  ...(query.customer ? { customerId: query.customer } : {}),
  ...(query.platform ? { providerIdentifier: query.platform } : {}),
},
deletedAt: null,
parentPostId: null,
```

In `getPostsList()` (around line 238-244), similarly replace the spread pattern. The current where clause doesn't have an explicit `integration: { deletedAt: null }` but add the platform filter alongside customer:
```typescript
...(query.customer || query.platform
  ? {
      integration: {
        ...(query.customer ? { customerId: query.customer } : {}),
        ...(query.platform ? { providerIdentifier: query.platform } : {}),
      },
    }
  : {}),
```

The `providerIdentifier` field on the Integration model is already indexed (`@@index([providerIdentifier])` in schema.prisma), so filtering is efficient.
