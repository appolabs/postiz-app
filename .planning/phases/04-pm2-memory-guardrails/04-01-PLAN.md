# Context: What Exists Today

## Current PM2 Startup Chain

The Docker entrypoint (`var/docker/entrypoint.sh`) ends with:

```
export NODE_OPTIONS="${NODE_OPTIONS:---max-old-space-size=384}"
nginx && pnpm run pm2
```

Root `package.json` defines:
```json
"pm2": "pnpm run pm2-run",
"pm2-run": "pm2 delete all || true && pnpm run prisma-db-push && pnpm run --parallel pm2 && pm2 logs"
```

This fans out to three per-app `pm2` scripts that each call `pm2 start pnpm`:
- `apps/backend/package.json`:       `"pm2": "pm2 start pnpm --name backend -- start"`
- `apps/frontend/package.json`:      `"pm2": "pm2 start pnpm --name frontend -- start"`
- `apps/orchestrator/package.json`:  `"pm2": "pm2 start pnpm --name orchestrator -- start"`

Each app's `start` script runs via dotenv:
- **backend**: `dotenv -e ../../.env -- node --experimental-require-module ./dist/apps/backend/src/main.js`
- **frontend**: `dotenv -e ../../.env -- next start -p 4200`
- **orchestrator**: `dotenv -e ../../.env -- node --experimental-require-module ./dist/apps/orchestrator/src/main.js`

## What's Wrong

1. **No memory limits.** PM2 has `max_memory_restart` but it's unused. If any process leaks, it grows until the OS OOM-killer terminates the entire container (all 3 processes + Temporal).

2. **Indirect process tree.** `pm2 start pnpm --name backend -- start` spawns: pm2 -> pnpm -> dotenv -> node. PM2 monitors the pnpm wrapper, not the actual Node process. This means `max_memory_restart` would measure pnpm's RSS, not Node's. PM2 also can't send SIGTERM directly to Node.

3. **Scattered configuration.** PM2 process definitions live across 4 files (root + 3 apps). Adding a setting (like memory limits) requires editing all of them.

4. **NODE_OPTIONS heap cap (384MB) is the only guard.** Set in entrypoint.sh, it caps V8 old-space but doesn't account for native memory (buffers, C++ addons like sharp/canvas/bcrypt). A process can exceed 384MB total RSS while V8 stays under the heap cap.

## Memory Budget

Container: 2GB total
- Temporal server: ~200-300MB
- nginx: ~10MB
- OS + overhead: ~100MB
- Available for 3 Node processes: ~1.3-1.7GB
- Per-process target: ~400-500MB RSS max

With `--max-old-space-size=384` already set, V8 heap won't exceed 384MB, but total RSS (heap + native + buffers) can reach ~500MB. Setting `max_memory_restart: "450M"` gives headroom above the heap cap while catching native memory leaks before they reach the danger zone.

# Changes

## New File: `ecosystem.config.js` (repo root)

```js
// PM2 process configuration for Postiz single-container deployment.
// Started by: pm2 start ecosystem.config.js
// Memory limits prevent OOM on the 2GB container.

const path = require('path');
const dotenvPath = path.resolve(__dirname, '.env');

// Common env applied to all processes.
// NODE_OPTIONS is inherited from the entrypoint (--max-old-space-size=384).
// dotenv-cli is removed from the process chain — PM2 loads .env directly.
const commonEnv = {};

module.exports = {
  apps: [
    {
      name: 'backend',
      script: 'node',
      args: '--experimental-require-module ./dist/apps/backend/src/main.js',
      cwd: __dirname,
      node_args: '',
      max_memory_restart: '450M',
      env_file: dotenvPath,
      autorestart: true,
      watch: false,
      instances: 1,
      ...commonEnv,
    },
    {
      name: 'frontend',
      script: 'node_modules/.bin/next',
      args: 'start -p 4200',
      cwd: __dirname,
      max_memory_restart: '450M',
      env_file: dotenvPath,
      autorestart: true,
      watch: false,
      instances: 1,
      ...commonEnv,
    },
    {
      name: 'orchestrator',
      script: 'node',
      args: '--experimental-require-module ./dist/apps/orchestrator/src/main.js',
      cwd: __dirname,
      max_memory_restart: '450M',
      env_file: dotenvPath,
      autorestart: true,
      watch: false,
      instances: 1,
      ...commonEnv,
    },
  ],
};
```

Key design decisions:
- **Direct node/next invocation** instead of `pnpm start` wrapper. PM2 monitors the actual Node process, so `max_memory_restart` measures real RSS. This eliminates the pm2 -> pnpm -> dotenv -> node chain.
- **`env_file`** replaces `dotenv -e ../../.env --`. PM2 loads the .env file directly into each process's environment. The `dotenv-cli` wrapper is no longer needed in the PM2 path.
- **`cwd: __dirname`** ensures all processes run from the repo root (where `dist/` and `node_modules/` live), same as today.
- **450M limit** per process. V8 heap cap is 384MB, so total RSS (heap + native allocations) typically sits around 400-430MB. 450M gives ~20MB headroom above normal peak while catching leaks before they threaten the container.
- **`--experimental-require-module`** preserved for backend and orchestrator (required for ESM module loading in the built output).
- **Frontend uses `node_modules/.bin/next`** directly. In the Docker image, packages are installed at the repo root (`/app`), so the next binary is at `/app/node_modules/.bin/next`. This is equivalent to `pnpm exec next start -p 4200`.

# Risks

1. **`env_file` support.** PM2's `env_file` option was added in PM2 5.x. The Docker image installs PM2 via `npm --global install pm2` (latest). If the installed version is older, `env_file` silently does nothing and processes start without .env vars. **Mitigation:** Verify PM2 version in Docker build. If < 5.4.0, fall back to using the `dotenv` npm package inside ecosystem.config.js to parse .env and spread into `env: {}`.

2. **`node_modules/.bin/next` path for frontend.** In the Docker image, all packages are installed at `/app` (repo root). The next binary should be at `/app/node_modules/.bin/next`. If pnpm hoisting changes, the path could break. **Mitigation:** After building the Docker image, verify `ls /app/node_modules/.bin/next` exists. Alternative: use `npx next` or `pnpm exec next` as the script (but that re-introduces a wrapper process).

3. **450M too aggressive or too lenient.** If a process legitimately needs >450MB RSS (e.g., during a burst of image processing with sharp), PM2 will restart it, causing a brief outage for that service. If 450M is too high and all 3 hit it simultaneously (1.35GB), plus Temporal (300MB), that's 1.65GB — tight but within 2GB. **Mitigation:** Monitor RSS via `pm2 monit` after deploy. Adjust limits per-process if needed (frontend typically uses less than backend).

4. **NODE_OPTIONS inheritance.** The entrypoint exports `NODE_OPTIONS="--max-old-space-size=384"`. PM2 passes environment to child processes, so this should propagate. But if PM2's `env_file` or `env` overrides NODE_OPTIONS, the heap cap could be lost. **Mitigation:** ecosystem.config.js does not set NODE_OPTIONS, so the inherited value is preserved.

5. **Prisma schema push timing.** `prisma-db-push` runs before `pm2 start` in the pm2-run script. This is unchanged behavior but worth noting: if the DB is unreachable, the startup chain fails before PM2 starts. This is the same as today.
