# Phase 8: Multi-Stage Dockerfile

## 1. Context: Current Image Bloat

The current `Dockerfile.dev` is a single-stage build:

```
FROM temporalio/auto-setup:1.29.3 AS temporal   # only used for COPY --from
FROM node:22.20-bookworm-slim                    # single stage: build + run
  apt-get install g++ make python3-pip bash netcat-openbsd nginx
  npm install pnpm pm2
  COPY . /app
  pnpm install          # all deps (prod + dev)
  pnpm run build        # compiles backend, orchestrator, frontend
  COPY --from=temporal   # Temporal binaries
  CMD entrypoint.sh
```

**What stays in the final image that should not:**
- `g++`, `make`, `python3-pip` (~200-300MB) -- only needed to compile native addons
- All `devDependencies` in node_modules (~500MB+ of test frameworks, linters, bundlers,
  TypeScript compiler, Vite, Jest, ESLint, etc.)
- Source TypeScript files (only compiled JS is needed at runtime)
- Build caches, `.git`, etc.

**Current node_modules is ~3.1GB** (dev workstation, all platforms). In Docker (linux-only),
it will be smaller, but still contains all devDependencies.

## 2. Multi-Stage Design

### Stage 1: `temporal` (existing pattern, unchanged)

```dockerfile
FROM temporalio/auto-setup:1.29.3 AS temporal
```

Provides Temporal binaries:
- `/usr/local/bin/temporal`
- `/usr/local/bin/temporal-server`
- `/usr/local/bin/temporal-sql-tool`
- `/usr/local/bin/dockerize`
- `/etc/temporal/` (config templates, auto-setup.sh, schema SQL)

### Stage 2: `builder` -- Full build environment

Base: `node:22.20-bookworm-slim`

Purpose: Install ALL dependencies (including devDependencies), run the full build, then
prepare a production-only node_modules.

Steps:
1. Install build tools: `g++`, `make`, `python3-pip` (needed for native addons: bcrypt, canvas)
2. Install pnpm globally
3. Copy source, install all deps (`pnpm install`)
4. Run `pnpm run build` (compiles backend, orchestrator, frontend)
5. Prune dev dependencies: `pnpm prune --prod`

**Critical: pnpm prune --prod** removes all packages listed in `devDependencies` from
`node_modules`. This eliminates TypeScript, Jest, ESLint, Vite, and all other build-only
tools. The pruned node_modules contains only production dependencies.

### Stage 3: `runner` -- Slim production image

Base: `node:22.20-bookworm-slim`

Purpose: Minimal runtime with only what's needed to run the application.

**Install ONLY runtime system packages:**
- `bash` -- entrypoint.sh is a bash script
- `netcat-openbsd` -- used by entrypoint for TCP port checks
- `nginx` -- reverse proxy in front of backend + frontend
- Runtime libraries for native modules (see Section 4)

**DO NOT install:** `g++`, `make`, `python3-pip`

Copy from builder:
- Pruned `node_modules/` (production only)
- Build outputs (see Section 3)
- Package files needed by pnpm scripts (`package.json`, workspace configs)

Copy from temporal:
- Temporal binaries and config

Copy from repo:
- `var/docker/nginx.conf`
- `var/docker/entrypoint.sh`

Install pnpm + pm2 globally (needed at runtime for process management).

## 3. What Goes in the Runner Stage

### 3A. Backend + Orchestrator (NestJS)

NestJS compiles to plain JS. Build output structure:

- Backend: `apps/backend/dist/apps/backend/src/main.js` (and tree)
- Orchestrator: `apps/orchestrator/dist/apps/orchestrator/src/main.js` (and tree)

The start scripts are:
```
backend:      node --experimental-require-module ./dist/apps/backend/src/main.js
orchestrator: node --experimental-require-module ./dist/apps/orchestrator/src/main.js
```

These run from the workspace root via pnpm filter, so the `dist/` directories must be
at their original relative paths within `/app`.

**Required in runner:**
- `apps/backend/dist/` -- compiled backend JS
- `apps/orchestrator/dist/` -- compiled orchestrator JS
- `apps/backend/package.json` -- pnpm filter needs this
- `apps/orchestrator/package.json` -- pnpm filter needs this

### 3B. Frontend (Next.js) -- TWO SCENARIOS

#### Scenario A: Without Phase 5 (standalone) -- CURRENT STATE

`next.config.js` does NOT have `output: 'standalone'`. The frontend runs as:
```
next start -p 4200
```

This requires the full `.next/` build output AND the complete `node_modules` tree
(Next.js resolves dependencies at runtime). The `next` binary itself comes from
node_modules.

**Required in runner:**
- `apps/frontend/.next/` -- build output
- `apps/frontend/package.json` -- pnpm filter needs this
- `apps/frontend/next.config.js` -- runtime config (rewrites, redirects, headers)
- `apps/frontend/public/` -- static assets (if exists)
- Full `node_modules/` (already copied for backend/orchestrator, shared via hoisting)

#### Scenario B: With Phase 5 (standalone) -- FUTURE STATE

If Phase 5 adds `output: 'standalone'` to `next.config.js`, Next.js produces a
self-contained build at `.next/standalone/` with its own minimal `node_modules`.

**Required in runner (standalone):**
- `apps/frontend/.next/standalone/` -- self-contained server + minimal node_modules
- `apps/frontend/.next/static/` -- must be at `.next/static` relative to standalone server
- `apps/frontend/public/` -- static assets (if exists)

The standalone server runs as `node apps/frontend/.next/standalone/server.js` and does
NOT need the full node_modules tree for the frontend.

**Recommendation:** Implement for Scenario A now (no standalone). Phase 5 can optimize
the COPY commands later. The multi-stage structure supports both -- only the frontend
COPY lines change.

### 3C. Prisma Client

The Prisma generated client lives in `node_modules/.prisma/client/` and includes a
platform-specific query engine binary (`libquery_engine-debian-openssl-*.so.node` on Linux).

**Required in runner:**
- `node_modules/.prisma/client/` -- generated client with query engine
- `node_modules/@prisma/client/` -- the Prisma client package
- The schema file is embedded in the generated client, no separate copy needed

Prisma generation runs at `pnpm install` (via `postinstall` script). The generated client
is in the builder's node_modules and survives `pnpm prune --prod` (since `@prisma/client`
is a production dependency). The `.prisma/client/` generated output also survives pruning.

**prisma-db-push at runtime:** The `pm2-run` script calls `pnpm run prisma-db-push` which
uses `pnpm dlx prisma@6.5.0`. This downloads prisma CLI on first run. This is fine -- it
does not require build tools. However, the schema file must be accessible:
`libraries/nestjs-libraries/src/database/prisma/schema.prisma`

**Required in runner (for prisma-db-push):**
- `libraries/nestjs-libraries/src/database/prisma/schema.prisma`

### 3D. Temporal Binaries

Copied from the `temporal` stage (unchanged from current Dockerfile):
- `/usr/local/bin/temporal`
- `/usr/local/bin/temporal-server`
- `/usr/local/bin/temporal-sql-tool`
- `/usr/local/bin/dockerize`
- `/etc/temporal/` (config templates, auto-setup.sh, schema migrations)

### 3E. Nginx

- Install `nginx` package in runner stage
- Copy `var/docker/nginx.conf` to `/etc/nginx/nginx.conf`
- Create `www` user/group (for nginx worker processes)

### 3F. Entrypoint

- Copy `var/docker/entrypoint.sh` to `/app/entrypoint.sh`
- `chmod +x`

### 3G. Workspace Configuration Files

The `pnpm run pm2` command at the root needs:
- `package.json` (root) -- scripts, workspace config
- `pnpm-workspace.yaml` -- workspace packages definition
- `.npmrc` -- pnpm configuration (node-linker=hoisted)
- `pnpm-lock.yaml` -- pnpm needs this for `dlx` and other operations
- `ecosystem.config.js` -- if Phase 4 has been executed (pm2 config)

## 4. Native Module Handling

### Modules with Native Bindings (production dependencies)

| Module | Build mechanism | Runtime needs |
|--------|----------------|---------------|
| `sharp` | Uses prebuilt `@img/sharp-linux-x64` (optional dep, auto-selected) | `libvips` bundled inside the platform package. No system libs needed. |
| `bcrypt` | `node-pre-gyp install --fallback-to-build` downloads prebuilt binary. Fallback needs g++. | Prebuilt `.node` binary. No extra system libs. |
| `canvas` | `node-pre-gyp install --fallback-to-build` downloads prebuilt binary. Fallback needs g++, python, pkg-config, libcairo2-dev, etc. | Prebuilt binary needs runtime libs: `libcairo2`, `libjpeg62-turbo`, `libpango-1.0-0`, `libgif7`, `librsvg2-2`, `libpixman-1-0`. |
| `@temporalio/core-bridge` | Ships prebuilt binaries per platform in `releases/`. No compilation needed at install. | Prebuilt `.node` binary. Needs `libssl3` (included in bookworm-slim). |
| `bufferutil` / `utf-8-validate` | node-gyp build at install (needs g++). | Compiled `.node` binary. No extra system libs. |

### Strategy

**In builder stage:** Install `g++`, `make`, `python3-pip` so that native modules can
compile if prebuilt binaries are unavailable. Also install canvas runtime libs as build
dependencies (canvas build links against them).

**In runner stage:** Install ONLY the runtime shared libraries that native modules need:
```
apt-get install -y --no-install-recommends \
    libcairo2 \
    libjpeg62-turbo \
    libpango-1.0-0 \
    libgif7 \
    librsvg2-2 \
    libpixman-1-0
```

These are small shared libraries (~10-20MB total) vs the ~200-300MB of build toolchain.

**sharp** bundles its own libvips -- no system dependency needed.

**bcrypt** prebuilt binaries are statically linked -- no system dependency needed.

**@temporalio/core-bridge** ships prebuilt for `x86_64-unknown-linux-gnu` and
`aarch64-unknown-linux-gnu`. It links against libssl which is in bookworm-slim.

### canvas Runtime Libraries

The `canvas` npm package (node-canvas) uses node-pre-gyp to download prebuilt binaries
for the platform. These binaries are dynamically linked against system Cairo/Pango/etc.
The runner stage MUST have these shared libraries installed or canvas will fail at runtime
with `Error: libcairo.so.2: cannot open shared object file`.

To verify which libs are needed, run in the builder:
```
ldd node_modules/canvas/build/Release/canvas.node
```

## 5. pnpm Prune Strategy

### How `pnpm prune --prod` works

With `node-linker=hoisted` (as configured in `.npmrc`), pnpm creates a flat node_modules
similar to npm. `pnpm prune --prod` removes all packages that are only in `devDependencies`.

**What gets removed (~500MB+ estimated):**
- TypeScript (`typescript`)
- Bundlers (`vite`, `tsup`, `@swc/*`)
- Test frameworks (`jest`, `vitest`, `@testing-library/*`)
- Linters (`eslint`, `prettier`, all eslint plugins)
- NestJS CLI + schematics (`@nestjs/cli`, `@nestjs/schematics`)
- Type definitions in devDeps (`@types/jest`, `@types/chrome`, etc.)
- Build plugins (`@vitejs/plugin-react`, `@crxjs/vite-plugin`)
- Dev utilities (`nodemon`, `cross-env`)

**What stays (production dependencies):**
- All packages in `dependencies` (NestJS runtime, Next.js, Prisma client, sharp, etc.)
- Their transitive dependencies
- Native addon binaries (bcrypt, canvas, sharp platform packages)
- Prisma generated client

### Pruning Sequence

```dockerfile
# In builder stage, after build completes:
RUN pnpm prune --prod
```

This must run AFTER `pnpm run build` because the build needs devDependencies (TypeScript
compiler, NestJS CLI, etc.).

**Important:** `pnpm prune --prod` modifies node_modules in-place. After pruning, the
builder's node_modules contains only production packages. This is what gets copied to
the runner stage.

### pnpm Store Cleanup

The pnpm content-addressable store (`.pnpm-store`) is NOT in node_modules and is not
copied. No extra cleanup needed.

## 6. Dockerfile Implementation

```dockerfile
# =============================================================================
# Stage 1: Temporal binaries
# =============================================================================
FROM temporalio/auto-setup:1.29.3 AS temporal

# =============================================================================
# Stage 2: Builder -- install deps, build, prune
# =============================================================================
FROM node:22.20-bookworm-slim AS builder

ARG NEXT_PUBLIC_VERSION
ENV NEXT_PUBLIC_VERSION=$NEXT_PUBLIC_VERSION
ARG NEXT_PUBLIC_BACKEND_URL
ENV NEXT_PUBLIC_BACKEND_URL=$NEXT_PUBLIC_BACKEND_URL

# Build tools for native addons + canvas build dependencies
RUN apt-get update && apt-get install -y --no-install-recommends \
    g++ \
    make \
    python3-pip \
    libcairo2-dev \
    libjpeg62-turbo-dev \
    libpango1.0-dev \
    libgif-dev \
    librsvg2-dev \
    libpixman-1-dev \
&& rm -rf /var/lib/apt/lists/*

RUN npm --no-update-notifier --no-fund --global install pnpm@10.6.1

WORKDIR /app
COPY . /app

RUN pnpm install
RUN NODE_OPTIONS="--max-old-space-size=4096" pnpm run build

# Remove dev dependencies -- only production packages remain
RUN pnpm prune --prod

# =============================================================================
# Stage 3: Runner -- slim production image
# =============================================================================
FROM node:22.20-bookworm-slim AS runner

# Runtime-only system packages
RUN apt-get update && apt-get install -y --no-install-recommends \
    bash \
    netcat-openbsd \
    nginx \
    libcairo2 \
    libjpeg62-turbo \
    libpango-1.0-0 \
    libgif7 \
    librsvg2-2 \
    libpixman-1-0 \
&& rm -rf /var/lib/apt/lists/*

# nginx user
RUN addgroup --system www \
 && adduser --system --ingroup www --home /www --shell /usr/sbin/nologin www \
 && mkdir -p /www \
 && chown -R www:www /www /var/lib/nginx

# Runtime tools (pnpm for scripts, pm2 for process management)
RUN npm --no-update-notifier --no-fund --global install pnpm@10.6.1 pm2

WORKDIR /app

# --- Copy production node_modules (pruned) ---
COPY --from=builder /app/node_modules ./node_modules

# --- Copy build outputs ---
# Backend
COPY --from=builder /app/apps/backend/dist ./apps/backend/dist
COPY --from=builder /app/apps/backend/package.json ./apps/backend/package.json

# Orchestrator
COPY --from=builder /app/apps/orchestrator/dist ./apps/orchestrator/dist
COPY --from=builder /app/apps/orchestrator/package.json ./apps/orchestrator/package.json

# Frontend (Scenario A: no standalone -- full .next + config)
COPY --from=builder /app/apps/frontend/.next ./apps/frontend/.next
COPY --from=builder /app/apps/frontend/package.json ./apps/frontend/package.json
COPY --from=builder /app/apps/frontend/next.config.js ./apps/frontend/next.config.js

# --- Copy workspace config files ---
COPY --from=builder /app/package.json ./package.json
COPY --from=builder /app/pnpm-workspace.yaml ./pnpm-workspace.yaml
COPY --from=builder /app/pnpm-lock.yaml ./pnpm-lock.yaml
COPY --from=builder /app/.npmrc ./.npmrc

# --- Copy Prisma schema (needed by prisma-db-push at runtime) ---
COPY --from=builder /app/libraries/nestjs-libraries/src/database/prisma/schema.prisma \
     ./libraries/nestjs-libraries/src/database/prisma/schema.prisma

# --- Copy ecosystem.config.js if Phase 4 has been executed ---
# COPY --from=builder /app/ecosystem.config.js ./ecosystem.config.js

# --- Temporal binaries ---
COPY --from=temporal /usr/local/bin/temporal /usr/local/bin/
COPY --from=temporal /usr/local/bin/temporal-server /usr/local/bin/
COPY --from=temporal /usr/local/bin/temporal-sql-tool /usr/local/bin/
COPY --from=temporal /usr/local/bin/dockerize /usr/local/bin/
COPY --from=temporal /etc/temporal/ /etc/temporal/

# --- Nginx config ---
COPY var/docker/nginx.conf /etc/nginx/nginx.conf

# --- Entrypoint ---
COPY var/docker/entrypoint.sh /app/entrypoint.sh
RUN chmod +x /app/entrypoint.sh

CMD ["/app/entrypoint.sh"]
```

## 9. Risks and Mitigations

### Risk 1: Missing runtime shared libraries for canvas

**Symptom:** `Error: libcairo.so.2: cannot open shared object file` at runtime.

**Mitigation:** The runner stage installs `libcairo2`, `libjpeg62-turbo`, `libpango-1.0-0`,
`libgif7`, `librsvg2-2`, `libpixman-1-0`. If canvas has additional deps, use
`ldd node_modules/canvas/build/Release/canvas.node` in builder to discover them.

**Fallback:** If canvas prebuilt binary is not available for the platform, the builder stage
has g++ and build tools to compile from source. The compiled `.node` file is in node_modules
and survives `pnpm prune --prod`.

### Risk 2: pnpm prune removes something needed at runtime

**Symptom:** `MODULE_NOT_FOUND` error when starting a process.

**Mitigation:** All runtime-needed packages should be in `dependencies`, not
`devDependencies`. Currently, some `@types/*` packages are in `dependencies` (which is
unusual but harmless -- they add type definitions but no runtime code). These survive
pruning.

**Specific concern:** `dotenv-cli` is in `dependencies` and is used by start scripts
(`dotenv -e ../../.env -- ...`). It will survive pruning. Good.

**Specific concern:** `cross-env` is in `devDependencies` and is used by build scripts
only. After pruning, build scripts cannot run -- but they don't need to. Good.

### Risk 3: Prisma CLI not available for db-push at runtime

**Symptom:** `prisma-db-push` script fails because prisma is in devDependencies.

**Mitigation:** The script uses `pnpm dlx prisma@6.5.0` which downloads the prisma CLI
on demand. It does NOT depend on the locally installed prisma package. The schema file
must be at the expected path -- we copy it explicitly.

### Risk 4: node_modules COPY is slow and large

**Symptom:** Large Docker layer for node_modules, slow builds.

**Mitigation:** Even after pruning, production node_modules is likely ~1-1.5GB. This is
inherent to the project's dependency tree (200+ production packages). The savings come
from removing devDependencies (~500MB+) and build tools (~200-300MB).

**Future optimization:** Phase 5 (Next.js standalone) would allow NOT copying the entire
node_modules for the frontend, reducing the runtime node_modules further.

### Risk 5: .dockerignore excludes needed files

**Symptom:** Build fails because source files are missing in builder.

**Mitigation:** Check `.dockerignore` (if it exists). The current Dockerfile uses
`COPY . /app` which copies everything not in `.dockerignore`. Ensure `node_modules`,
`.git`, and `dist` are in `.dockerignore` (they should not be copied into builder --
builder creates its own).

### Risk 6: Build cache invalidation

**Symptom:** Every code change re-installs all dependencies.

**Mitigation:** Optimize COPY order in builder stage. Copy `package.json`,
`pnpm-lock.yaml`, `pnpm-workspace.yaml`, `.npmrc`, and all workspace `package.json`
files FIRST, then `pnpm install`, then copy source. This way dependency installation
is cached unless lockfile changes.

```dockerfile
# In builder stage -- optimized for Docker cache:
COPY package.json pnpm-lock.yaml pnpm-workspace.yaml .npmrc ./
COPY apps/backend/package.json ./apps/backend/
COPY apps/frontend/package.json ./apps/frontend/
COPY apps/orchestrator/package.json ./apps/orchestrator/
COPY apps/cli/package.json ./apps/cli/
COPY apps/sdk/package.json ./apps/sdk/
COPY apps/commands/package.json ./apps/commands/
COPY apps/extension/package.json ./apps/extension/
COPY libraries/helpers/package.json ./libraries/helpers/
COPY libraries/nestjs-libraries/package.json ./libraries/nestjs-libraries/
COPY libraries/react-shared-libraries/package.json ./libraries/react-shared-libraries/
COPY libraries/plugins/package.json ./libraries/plugins/

# Prisma schema needed by postinstall (prisma-generate)
COPY libraries/nestjs-libraries/src/database/prisma/schema.prisma \
     ./libraries/nestjs-libraries/src/database/prisma/schema.prisma

RUN pnpm install

# NOW copy all source
COPY . .
RUN NODE_OPTIONS="--max-old-space-size=4096" pnpm run build
RUN pnpm prune --prod
```

This is a significant optimization -- dependency installation is the slowest step (~2-5min)
and should only re-run when dependencies change.
